# EXP-5: 动态策略热更新效果验证实验

**结果位置**: 本实验的结果保存在 `results/exp5/` 和 `analysis/figures/` 目录下
- `results/exp5/dynamic_policy_effect.txt` - 动态策略效果数据
- `results/exp5/exp5_hot_update_effect.png` - 热更新效果可视化
- `analysis/figures/exp5_hot_update_effect.png` - 策略更新延迟分析图

## 实验目标

验证拦截系统的动态策略热更新机制，评估策略变更的响应时间和生效效果。

**核心问题**: 当管理员动态调整租户配额时，拦截系统能否及时生效？策略下发到实际生效的延迟是多少？

## 实验方法

### 实验设计

| 参数 | 配置 |
|------|------|
| **租户ID** | 50 |
| **初始配额** | 5 QP |
| **更新后配额** | 10 QP |
| **更新时机** | 第7次创建尝试 |
| **创建间隔** | 500 ms |
| **总尝试次数** | 20次 |

### 测试流程

```

0s    2.5s    3.5s    5s    7.5s    10s
│      │       │       │      │       │
▼      ▼       ▼       ▼      ▼       ▼
┌──────┬───────┬───────┬──────┬───────┐
│ 创建 │  创建 │  创建 │ 创建 │  创建 │
│ QP1  │  QP5  │  QP7  │ QP10 │ QP20  │
```

### 关键指标

| 指标 | 描述 | 计算方法 |
|------|------|----------|
| **策略下发延迟** | 从执行更新命令到生效的时间 | `POLICY_EFFECTIVE_TIME - POLICY_UPDATE_TIME` |
| **更新前成功率** | 配额更新前的QP创建成功率 | `BEFORE_UPDATE_SUCCESS / BEFORE_UPDATE_ATTEMPTS` |
| **更新后成功率** | 配额更新后的QP创建成功率 | `AFTER_UPDATE_SUCCESS / AFTER_UPDATE_ATTEMPTS` |
| **总成功率** | 全部尝试的成功率 | `TOTAL_SUCCESS / TOTAL_ATTEMPTS` |

### 策略下发延迟分析

| 指标 | 时间戳 | 说明 |
|------|--------|------|
| **策略下发时间** | 2503.595 ms | 管理员执行更新命令 |
| **策略生效时间** | 3004.525 ms | 应用程序可创建更多QP |
| **策略延迟** | **500.930 ms** | 从下发到生效的延迟 |
| **首次成功** | 第7次尝试 | 更新后首次成功创建 |

### 关键发现

**1. 配额更新成功生效**
- 初始配额5 QP，成功创建5个QP
- 更新后配额10 QP，又成功创建5个QP（共10个）
- 证明热更新机制工作正常

**2. 策略延迟约500ms**
- 从执行更新命令到生效约500ms
- 延迟来源：
  - 共享内存同步时间
  - 拦截库读取新配额的时间
  - 应用程序下次尝试创建QP的时间（500ms间隔）

**3. 配额严格执行**
- 成功创建10个QP后，后续10次尝试均失败
- 证明新配额被严格执行

## 实验结论

### 主要发现

1. **热更新机制有效**
   - 配额从5 QP动态调整到10 QP成功生效
   - 应用程序无需重启即可感知新配额
   - 更新前后配额执行严格

2. **策略延迟约500ms**
   - 从下发到生效的延迟约500ms
   - 延迟主要来源于共享内存同步和应用程序轮询间隔
   - 对于大多数场景，500ms延迟可接受

3. **动态调整灵活**
   - 可根据负载情况实时调整配额
   - 支持扩容（增加配额）和缩容（减少配额）
   - 不影响正在运行的应用程序

### 实际意义

本实验验证了动态策略热更新的有效性：
- ✅ **热更新支持**: 无需重启即可调整配额
- ✅ **响应及时**: 500ms内生效，满足大多数场景
- ✅ **灵活调度**: 支持根据负载动态调整资源分配

### 与预期的对比

| 指标 | 预期 | 实际 | 评价 |
|------|------|------|------|
| 热更新生效 | 是 | 是 | ✅ 达标 |
| 策略延迟 | <1s | ~500ms | ✅ 达标 |
| 配额执行 | 严格 | 严格 | ✅ 达标 |

## 快速开始

### 手动运行实验

```bash
cd /home/why/rdma_intercept_ldpreload/experiments/exp5_dynamic_policy

# 1. 启动租户管理守护进程
../../build/tenant_manager_daemon --daemon --foreground &

# 2. 创建租户（初始配额=5）
../../build/tenant_manager_client create 50 5 100 "EXP5"

# 3. 编译测试程序
gcc -O2 -o build/exp5_dynamic_policy_effect src/exp5_dynamic_policy_effect.c -libverbs

# 4. 运行测试（终端1）
export LD_PRELOAD=../../build/librdma_intercept.so
export RDMA_INTERCEPT_ENABLE=1
export RDMA_TENANT_ID=50
export RDMA_INTERCEPT_ENABLE_QP_CONTROL=1
./build/exp5_dynamic_policy_effect results/exp5/test.txt

# 5. 在第7次尝试时（约3.5秒），在另一个终端执行更新
../../build/tenant_manager_client update 50 10 100

# 6. 查看结果
cat results/exp5/dynamic_policy_effect.txt
```
