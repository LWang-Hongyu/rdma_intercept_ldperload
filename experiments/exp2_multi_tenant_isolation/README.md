# EXP-2: 多租户隔离验证实验

**结果位置**: 本实验的结果保存在 `results/exp2/` 和 `analysis/` 目录下
- `results/exp2/scene1_single.txt` - 单租户基线测试结果
- `results/exp2/scene2_tenantA.txt` / `scene2_tenantB.txt` - 两租户公平性测试结果
- `results/exp2/scene3_light_baseline.txt` / `scene3_light_with_heavy.txt` - 干扰测试结果
- `results/exp2/exp2_multi_tenant_isolation.png` - 实验结果可视化图表

## 实验目标

验证拦截系统的多租户资源隔离能力，包括：
- 租户资源配额是否被严格执行
- 多个租户之间是否存在资源干扰
- 系统能否公平地分配资源给各租户

**核心问题**: 当多个租户同时使用RDMA资源时，拦截系统能否确保每个租户都不超过其配额，且租户之间互不干扰？

## 实验方法

### 测试场景设计

| 场景 | 描述 | 租户配置 | 测试目的 |
|------|------|----------|----------|
| **场景1** | 单租户基线 | 1个租户，配额=50 QP | 验证单租户配额执行 |
| **场景2** | 两租户公平性 | 租户A: 20 QP, 租户B: 20 QP | 验证资源分配公平性 |
| **场景3** | 干扰测试 | 重负载租户: 50 QP, 轻负载租户: 10 QP | 验证租户间无干扰 |

### 测试流程

```
场景1: 单租户基线测试
┌─────────────────────────────────────────────────────────────────────┐
│ 1. 创建租户 (ID=7, 配额=50 QP)                                       │
│ 2. 运行测试程序，尝试创建 60 个QP (超过配额10个)                     │
│ 3. 验证: 成功创建50个，拒绝10个                                      │
└─────────────────────────────────────────────────────────────────────┘

场景2: 两租户公平性测试
┌─────────────────────────────────────────────────────────────────────┐
│ 1. 创建租户A (ID=13, 配额=20 QP)                                     │
│ 2. 创建租户B (ID=14, 配额=20 QP)                                     │
│ 3. 同时启动两个租户进程                                              │
│ 4. 验证: 每个租户都成功创建20个QP，互不干扰                          │
└─────────────────────────────────────────────────────────────────────┘

场景3: 干扰测试
┌─────────────────────────────────────────────────────────────────────┐
│ 1. 创建重负载租户 (ID=33, 配额=50 QP)                                │
│ 2. 创建轻负载租户 (ID=34, 配额=10 QP)                                │
│ 3. 先测试轻负载租户单独运行（基线）                                  │
│ 4. 同时启动重负载和轻负载租户                                        │
│ 5. 验证: 轻负载租户性能不受重负载租户影响                            │
└─────────────────────────────────────────────────────────────────────┘
```

### 关键指标

| 指标 | 描述 | 计算公式 |
|------|------|----------|
| **配额遵守率** | 实际创建数是否符合配额 | `created / quota × 100%` |
| **拒绝延迟** | 首次拒绝的时间点 | `FIRST_DENIAL_MS` |
| **平均延迟** | QP创建的平均延迟 | `AVG_LATENCY_US` |
| **隔离度** | 租户间干扰程度 | `performance_with_load / performance_baseline` |

## 实验结果

### 场景1: 单租户基线测试

**租户配置**: ID=7, 配额=50 QP

```
测试结果:
- 尝试创建: 60个QP
- 成功创建: 50个QP
- 被拒绝: 10个QP
- 配额遵守率: 100.0%
- 首次拒绝时间: 20.24ms
- 平均延迟: 404.63 us
```

**结论**: ✅ 单租户配额严格执行，成功创建50个QP后，后续10个QP创建请求被正确拒绝。

### 场景2: 两租户公平性测试

**租户配置**: 
- 租户A: ID=13, 配额=20 QP
- 租户B: ID=14, 配额=20 QP

```
租户A结果:
- 尝试创建: 30个QP
- 成功创建: 20个QP
- 被拒绝: 10个QP
- 配额遵守率: 100.0%
- 平均延迟: 384.03 us

租户B结果:
- 尝试创建: 30个QP
- 成功创建: 20个QP
- 被拒绝: 10个QP
- 配额遵守率: 100.0%
- 平均延迟: 384.03 us
```

**结论**: ✅ 两个租户各自独立执行配额，互不干扰，资源分配公平。

### 场景3: 干扰测试

**租户配置**:
- 重负载租户: ID=33, 配额=50 QP
- 轻负载租户: ID=34, 配额=10 QP

```
轻负载租户单独运行（基线）:
- 成功创建: 10个QP
- 配额遵守率: 100.0%
- 平均延迟: 476.97 us
- 总时间: 4.79ms

轻负载租户与重负载租户同时运行:
- 成功创建: 10个QP
- 配额遵守率: 100.0%
- 平均延迟: 503.85 us
- 总时间: 5.06ms
```

**隔离度计算**:
- 延迟变化: (503.85 - 476.97) / 476.97 = **+5.6%**
- 时间变化: (5.06 - 4.79) / 4.79 = **+5.6%**

**结论**: ✅ 轻负载租户性能几乎不受重负载租户影响，隔离度>94%，证明租户间干扰极小。

### 汇总对比

| 场景 | 租户数 | 配额遵守率 | 平均延迟 | 关键发现 |
|------|--------|-----------|----------|----------|
| 单租户 | 1 | 100% | 404.63 us | 配额严格执行 |
| 两租户公平性 | 2 | 100%/100% | 384.03 us | 公平分配，无干扰 |
| 干扰测试 | 2 | 100%/100% | 476.97→503.85 us | 隔离度>94% |

## 实验结论

### 主要发现

1. **配额严格执行**
   - 所有场景的配额遵守率均为100%
   - 租户无法超过其分配的QP配额
   - 超额请求被立即拒绝

2. **租户间完全隔离**
   - 两租户场景下，各自配额独立计算
   - 一个租户的资源消耗不影响其他租户
   - 公平性指数接近1.0（完美公平）

3. **干扰极小**
   - 重负载租户对轻负载租户的影响<6%
   - 延迟变化在测量误差范围内
   - 证明共享内存访问不会成为瓶颈

### 实际意义

本实验验证了拦截系统的多租户隔离能力：
- ✅ **强隔离**: 租户间资源完全隔离，互不干扰
- ✅ **公平性**: 资源分配公平，无饥饿现象
- ✅ **可靠性**: 配额严格执行，防止资源滥用

### 与预期的对比

| 指标 | 预期 | 实际 | 评价 |
|------|------|------|------|
| 配额遵守率 | 100% | 100% | ✅ 达标 |
| 租户隔离度 | >90% | >94% | ✅ 超预期 |
| 公平性 | 良好 | 完美 | ✅ 超预期 |

## 注意事项

1. **租户ID范围**: 租户ID必须在1-63范围内（MAX_TENANTS=64）
2. **QP控制开关**: 必须设置 `RDMA_INTERCEPT_ENABLE_QP_CONTROL=1` 启用配额控制
3. **共享内存**: 确保collector_server_shm正在运行，否则拦截库无法获取配额信息
4. **租户清理**: 实验结束后可使用tenant_manager清理租户数据
