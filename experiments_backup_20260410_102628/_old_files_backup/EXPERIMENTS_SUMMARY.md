## EXP-1: 微基准测试 - 拦截开销评估

### 核心问题
LD_PRELOAD拦截机制引入的性能开销是多少？

### 实验方法
- **基线测试**: 直接调用原始RDMA API
- **拦截测试**: 通过LD_PRELOAD加载拦截库
- **对比指标**: 延迟差异百分比

### 测试变量
| 变量 | 类型 | 取值 |
|------|------|------|
| 操作类型 | 自变量 | QP_CREATE, QP_DESTROY, MR_REG, MR_DEREG |
| 迭代次数 | 控制变量 | 10,000次 |
| 延迟 | 因变量 | 微秒(μs) |

### 关键代码
```c
// 高精度计时
static inline double get_time_us() {
    struct timespec ts;
    clock_gettime(CLOCK_MONOTONIC, &ts);
    return ts.tv_sec * 1e6 + ts.tv_nsec / 1e3;
}

// 计算开销
overhead = (intercept_latency - baseline_latency) / baseline_latency * 100%
```

### 预期结果
| 操作 | 基线 | 拦截后 | 开销 |
|------|------|--------|------|
| QP_CREATE | 15μs | 18μs | 20% |
| MR_REG | 20μs | 23μs | 15% |

### 论文贡献点
1. 量化了LD_PRELOAD拦截的开销
2. 证明了开销在可接受范围内(<20%)
3. 为系统设计提供了性能依据

## EXP-2: 多租户隔离验证

### 核心问题
系统能否有效隔离不同租户的资源使用？

### 实验方法
- **公平性测试**: 两个相同配额租户并发运行
- **干扰测试**: 高负载租户对低负载租户的影响
- **配额测试**: 验证配额限制是否严格执行

### 测试场景
| 场景 | 租户A配额 | 租户B配额 | 测试目的 |
|------|----------|----------|---------|
| 公平性 | 20 | 20 | 资源分配公平性 |
| 干扰A | 100 | 10 | A满载对B的影响 |
| 干扰B | 10 | 100 | B满载对A的影响 |

### 关键指标
```python
# Jain's Fairness Index (1.0 = 完美公平)
fairness = (sum(allocations) ** 2) / (n * sum(a**2 for a in allocations))

# 隔离度 (1.0 = 完全隔离)
isolation = performance_with_load / performance_alone
```

### 预期结果
```
Fairness Index: 0.98 (> 0.9 PASS)
Isolation Level: 0.975 (> 0.95 PASS)
Quota Compliance: 100% (created <= quota)
```

### 论文贡献点
1. 证明了多租户隔离的有效性
2. 量化了隔离度(>97.5%)
3. 验证了配额机制的严格执行

## EXP-3: 可扩展性测试

### 核心问题
系统能支持多少并发租户？性能如何随规模变化？

### 实验方法
- **规模测试**: 1/5/10/20/50/100租户并发
- **压力测试**: 所有租户同时满负载运行
- **瓶颈分析**: 识别性能瓶颈

### 测试规模
```
租户数:     1    5    10   20   50   100
每租户QP:   100  20   10   5    2    1
总QP:       100  100  100  100  100  100
```

### 性能模型
```
吞吐量(T) = T_max / (1 + α*(N-1))
其中:
  T_max = 单租户最大吞吐量
  N = 租户数
  α = 竞争系数 (~0.01)
```

### 预期结果
```
1 Tenant:   40,000 ops/sec (Efficiency: 100%)
10 Tenants: 35,000 ops/sec (Efficiency: 87%)
50 Tenants: 25,000 ops/sec (Efficiency: 62%)
```

### 论文贡献点
1. 确定了系统的可扩展边界(50租户内高效)
2. 建立了性能与规模的数学模型
3. 识别了共享内存锁竞争瓶颈

## EXP-4: 缓存性能评估

### 核心问题
本地缓存机制能带来多少性能提升？

### 实验方法
- **命中率测试**: 不同访问模式的缓存命中率
- **延迟对比**: 缓存访问 vs 共享内存访问
- **自适应测试**: 固定TTL vs 自适应TTL

### 访问模式
| 模式 | 描述 | 预期命中率 |
|------|------|-----------|
| 顺序 | 重复访问同一资源 | 100% |
| 时间局部性 | 短时间重复访问 | >95% |
| 随机 | 随机访问不同资源 | ~85% |

### 关键指标
```
加速比 = SharedMemory_latency / Cache_latency
       = 100ns / 10ns
       = 10x
```

### 预期结果
```
Hit Rate (sequential): 100%
Hit Rate (random): 85.5%
Access Latency (cache): 10ns
Access Latency (shared memory): 100ns
Speedup: 10x
```

### 论文贡献点
1. 证明了缓存机制的有效性(10x加速)
2. 验证了自适应TTL调整(+7%命中率)
3. 为优化设计提供了数据支撑

## EXP-5: 动态策略效果评估

### 核心问题
动态策略能否比静态策略更好地分配资源？

### 实验方法
- **热更新测试**: 策略变更的响应延迟
- **自适应测试**: 负载变化时的自动调整
- **公平性对比**: 静态配额 vs 动态配额

### 动态调整算法
```python
if usage_rate > high_watermark:
    quota *= 1.2  # 增加20%
elif usage_rate < low_watermark:
    quota *= 0.9  # 减少10%
```

### 对比维度
| 维度 | 静态策略 | 动态策略 |
|------|---------|---------|
| 配置方式 | 固定配额 | 自动调整 |
| 资源利用率 | 75% | 88% |
| 公平性指数 | 0.75 | 0.92 |
| 响应速度 | N/A | <1ms |

### 预期结果
```
Policy Update Latency: 0.5ms
Fairness Improvement: +17%
Utilization Improvement: +13%
```

### 论文贡献点
1. 证明了动态策略的优越性
2. 量化了自适应调整的效果
3. 展示了热更新的低延迟特性
